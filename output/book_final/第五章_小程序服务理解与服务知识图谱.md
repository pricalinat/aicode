# 第5章：小程序服务理解与服务知识图谱——服务能力的认知

> **本章导读**
> 
> 服务是小程序生态的核心。当用户打开一个小程序时，他们不是在购买一个具体的商品，而是在获取一种能力——点外卖的能力、叫车的能力、办事的能力。这种"能力"如何被理解、被分类、被评估、被推荐，就是本章要解决的问题。
>

## 5.1 引言

### 5.1.1 为什么要理解服务？

在展开技术细节之前，我们需要先回答一个根本问题：为什么服务理解如此重要？

### 5.1.2 服务与商品的本质差异

理解服务之前，先理解商品。

商品是有形或无形的实体，但边界是清晰的。一部iPhone 15 Pro Max是商品，你可以描述它的屏幕尺寸、处理器型号、电池容量——这些都是可量化的属性。商品的"理解"本质上是属性提取和知识组织。

服务完全不同。服务是一种**能力**或**体验**，它没有物理边界。

举例来说，"家政服务"是什么？是打扫卫生？是做饭？是照顾老人？还是带孩子？严格来说，"家政服务"是一类能力的统称，具体服务取决于具体提供者。当一个小程序声称提供"家政服务"时，它到底提供什么？不看详细内容，你无法知道。

这就是服务理解的第一个核心挑战：**服务的抽象性**。

### 5.1.3 服务理解的核心问题

让我们形式化地定义服务理解问题：

**给定**：一个待理解的服务实体 $s$（包含名称、描述、截图、用户评价等信息）
**求解**：服务的结构化知识表示 $K(s)$

这个 $K(s)$ 应该回答以下问题：
- 这是什么类型的服务？
- 它能做什么？
- 适合什么人使用？
- 质量怎么样？
- 与其他服务是什么关系？

这四个问题对应服务理解的四个核心能力：

| 核心能力 | 回答的问题 | 技术对应 |
|---------|------------|---------|
| 服务分类 | 这是什么类型的服务？ | 文本分类、多标签学习 |
| 意图理解 | 用户想要什么服务？ | 意图识别、槽位填充 |
| 服务内容理解 | 这个服务具体提供什么？ | 实体识别、属性抽取、多模态理解 |
| 服务质量评估 | 这个服务好吗？ | 回归、排序学习预测 |

### 5.1.4 服务理解的独特挑战

为什么这些能力在服务场景下特别困难？

**挑战一：语义边界模糊**

商品分类通常有清晰的边界——手机是手机，服装是服装。但服务经常"跨界"：

```
"上门做饭" → 家政服务？餐饮服务？还是健康服务？
"共享办公" → 办公服务？商业地产？还是社交服务？
"亲子餐厅" → 餐饮服务？亲子服务？还是活动场所？
```

一个服务可能同时属于多个类目，传统的单标签分类无法处理。

**挑战二：表达方式多样**

同一种服务，服务提供者可能用完全不同的话术：

```
"专业保洁" vs "深度清洁" vs "家政小时工" vs "扫除专家"
"附近美食" vs "好吃推荐" vs "餐饮导航" vs "寻味之旅"
```

这些表达之间的语义关联，需要模型具备强大的语义理解能力。

**挑战三：多模态信息复杂**

服务的内容呈现方式远比商品丰富：
- 商品：一张主图 + 几张详情图 + 规格参数
- 服务：名称（几字）+ 描述（营销文案）+ 截图（UI界面）+ 用户评价（主观反馈）

服务的价值往往体现在"使用体验"上，而体验是难以用文字描述的。

**挑战四：动态变化频繁**

小程序可以快速迭代更新。据统计，主流小程序平台的月度更新率约为15%-20%，意味着每个月有超过六分之一的服务会发生内容变化。理解系统需要具备持续学习的能力。

## 5.2 服务分类：回答"这是什么服务"

### 5.2.1 服务分类解决什么问题？

当你面对百万级小程序服务时，第一件要做的事是什么？

**分类**。

用户说"我想点外卖"，系统需要知道哪些服务属于"外卖"类目；用户说"我想打车"，系统需要召回"出行"类目下的服务。没有准确的分类，后续的匹配和推荐都无从谈起。

服务分类的本质是：**建立服务与类目之间的映射关系**。

### 5.2.2 服务分类体系设计

好的分类体系是服务分类的基础。我们从三个维度设计分类：

**维度一：按服务性质**

| 一级类目 | 二级类目 | 示例服务 |
|---------|---------|---------|
| 工具服务 | 查询类、计算类、转换类 | 天气查询、计算器、翻译 |
| 交易服务 | 外卖、电商、票务、出行 | 美团外卖、京东购物、12306 |
| 内容服务 | 资讯、短视频、阅读、音乐 | 今日头条、抖音、爱奇艺 |
| 社交服务 | 通讯、社区、论坛 | 微信、知乎、豆瓣 |
| 游戏服务 | 休闲、竞技、益智 | 开心消消乐、王者荣耀 |

**维度二：按行业领域**

生活服务、健康医疗、教育培训、出行旅游、政务服务、媒体娱乐……这一体系与现实社会的行业划分保持一致，便于用户理解。

**维度三：按用户需求**

- 信息获取需求：查询类工具、资讯内容
- 生活便利需求：外卖、打车、家政
- 娱乐消费需求：游戏、视频、直播
- 社交互动需求：通讯、社区、论坛

需求导向的分类更贴近用户的实际使用场景。

### 5.2.3 服务分类的技术演进

服务分类技术经历了三个阶段的演进：

**第一阶段：规则匹配**

```python
# 伪代码示例
def classify_service(service_name, service_desc):
    keywords = {
        '外卖': ['外卖', '配送', '订餐', '餐饮'],
        '出行': ['打车', '叫车', '乘车', '公交', '地铁'],
        '电商': ['购物', '商城', '买', '卖']
    }
    
    text = service_name + service_desc
    for category, words in keywords.items():
        if any(w in text for w in words):
            return category
    return '其他'
```

优点：简单直接，解释性强
缺点：无法处理表达多样性，召回率低

**第二阶段：传统机器学习**

```python
# 使用TF-IDF + SVM
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.svm import LinearSVC

# 提取文本特征
vectorizer = TfidfVectorizer(max_features=10000)
X_train = vectorizer.fit_transform(train_texts)

# 训练分类器
clf = LinearSVC()
clf.fit(X_train, train_labels)

# 预测
X_test = vectorizer.transform(test_texts)
predictions = clf.predict(X_test)
```

优点：泛化能力增强，能处理一定的表达多样性
缺点：特征工程依赖性强，语义理解能力有限

**第三阶段：深度学习（当前主流）**

```python
# BERT微调分类
from transformers import BertForSequenceClassification

model = BertForSequenceClassification.from_pretrained(
    'bert-base-chinese',
    num_labels=50  # 50个服务类目
)

# 前向传播
outputs = model(
    input_ids=input_ids,
    attention_mask=attention_mask,
    labels=label
)
loss = outputs.loss
logits = outputs.logits
```

预训练语言模型（BERT、RoBERTa、ERNIE）能够理解深层语义，在服务分类任务上取得了显著效果。

### 5.2.4 层次化分类策略

服务类目通常具有层次结构（类目树），如何有效利用这种层次？

**方法一：扁平分类**

忽略层次，直接训练一个多分类模型。简单但可能丢失类目间的关联信息。

**方法二：层级分类（Hierarchical Classification）**

自顶向下逐层预测：

```
Layer 1: 交易服务 → 内容服务 → 工具服务 → ...
    ↓
Layer 2: 外卖 → 电商 → 票务 → ...
        ↓
Layer 3: 餐饮外卖 → 生鲜配送 → ...
```

每个层级使用独立的分类器，上一级预测结果作为下一级的输入约束。

**方法三：层次嵌入**

将类目层次结构编码为嵌入向量，使相似类目在向量空间中临近。分类时不仅预测类别，还预测类别在层次树中的位置。

### 5.2.5 实验数据与效果对比

| 方法 | 准确率 | 特点 |
|-----|-------|------|
| 规则匹配 | ~60% | 简单快速，适合冷启动 |
| TF-IDF + SVM | ~78% | 泛化能力一般 |
| BERT-base | ~92% | 效果最好，计算成本高 |
| BERT + 层次分类 | ~95% | 最佳效果，适合多级类目 |

> **注意**：以上数据为典型值，实际效果取决于类目体系的规模和数据的分布。

### 5.2.6 批判性分析：分类不是终点

服务分类看似基础，但存在几个深层问题：

**问题一：分类粒度的两难**

粗粒度分类（如"生活服务"）太宽，无法区分具体需求；细粒度分类（如"川菜外卖"、"湘菜外卖"）太细，标注成本高，且容易出现数据稀疏。

**问题二：多标签分类的真实需求**

一个服务往往属于多个类目。"亲子餐厅"既是餐饮服务，也是亲子服务。传统的单标签分类无法处理这种场景，需要多标签学习能力。

**问题三：动态类目的挑战**

新服务类型不断涌现（如2020年的社区团购），静态类目体系难以适应。系统需要具备类目扩展能力。

**问题四：分类与业务目标的脱节**

学术上的分类准确率不等于业务价值。一个"分类准确"的服务，如果放在错误的业务场景下，仍然无法产生价值。

## 5.3 意图理解：回答"用户想要什么"

### 5.3.1 意图理解的两层含义

"意图理解"在服务场景下有两层含义：

**用户侧**：用户在搜索或对话中表达的需求
**服务侧**：服务本身的功能意图

本节聚焦用户侧的意图理解。

### 5.3.2 意图识别的任务定义

**输入**：用户的自然语言查询 $q$（如"附近有什么好吃的川菜馆"）
**输出**：意图类型 $y$（如"查找餐厅"）

意图识别本质上是一个**文本分类**问题，但比普通分类更复杂：

1. **表达多样**：同一意图有数百种说法
   ```
   "想吃饭" = "找点吃的" = "有什么好吃的" = "推荐餐厅"
   ```

2. **多意图并存**：一个查询可能包含多个意图
   ```
   "推荐附近的川菜馆，要评分高的"
   → 意图1：查找餐厅
   → 意图2：筛选评分
   ```

3. **意图模糊**：需要上下文判断
   ```
   "苹果" 
   → 可能是水果（找水果店）
   → 可能是手机（找Apple授权店）
   → 可能是公司（找苹果零售店）
   ```

### 5.3.3 槽位填充：提取意图的细节

仅有意图类型不够，系统还需要知道意图的细节：

**输入**："帮我订一张明天北京到上海的高铁票"
**输出**：
- 意图：订票
- 槽位：
  - 出发地：北京
  - 目的地：上海
  - 日期：明天
  - 交通类型：高铁

槽位填充（Slot Filling）本质上是**序列标注**问题：

```python
# 输入：帮我订一张明天北京到上海的高铁票
# 标注：B-出发地 I-出发地 I-出发地 O O O O O B-目的地 ...

# BERT-CRF实现
from transformers import BertForTokenClassification

model = BertForTokenClassification.from_pretrained(
    'bert-base-chinese',
    num_labels=len(label_schema)  # BIO标注体系
)
```

### 5.3.4 意图识别与槽位填充的联合模型

实际系统中，意图识别和槽位填充不是独立的，而是联合建模：

**方案一：管道模型（Pipeline）**
```
查询 → 意图识别 → 槽位填充
```
优点：解耦简单，易于调试
缺点：错误可能传播（意图错了，槽位也会错）

**方案二：联合模型（Joint Model）**
```
查询 → 共享编码器 → 意图分类头
                    → 槽位标注头
```
优点：任务间信息共享，效果更好
缺点：模型更复杂，训练难度更高

**工业界实践**：微信小程序搜索的意图理解系统采用联合模型架构，结合预训练语言模型和注意力机制，实现了超过95%的意图识别准确率。

### 5.3.5 少样本与零样本意图识别

冷启动场景下，标注数据不足怎么办？

**提示学习（Prompt Learning）** 提供了一种解决方案：

```python
# 传统微调
query = "附近有什么好吃的"
model(query) → 分类标签

# 提示学习
query = "附近有什么好吃的"
prompt = "这是关于{mask}的查询"
model(query + prompt) → 预测mask位置的词
```

通过设计提示模板，可以让预训练语言模型在不Fine-tuning的情况下完成意图识别。

> **实验数据**：在仅使用5%标注数据的条件下，基于提示学习的方法可以达到全监督模型90%以上的效果（Li et al., 2023）。

### 5.3.6 批判性分析：意图理解的上限

**问题一：意图的动态演化**

用户意图不是静态的。新产品、新概念的出现会带来新的意图模式。"盲盒"一开始是什么意图？没有人能预判。系统需要持续学习的能力。

**问题二：意图与情感的交织**

"我想找个靠谱的外卖"——这是"查找外卖"的意图，还是"吐槽外卖"的情感？意图识别和情感分析的边界在哪里？

**问题三：多轮对话中的意图漂移**

多轮对话中，用户意图可能发生变化：
```
用户：帮我找个川菜馆
系统：推荐了老成都川菜馆
用户：不要太辣的（修改意图）
系统：需要重新理解
```

意图追踪（Intent Tracking）是开放问题。

## 5.4 服务内容理解：回答"服务具体提供什么"

### 5.4.1 服务内容的多模态本质

服务的内容呈现是多模态的：

| 模态 | 内容 | 理解难点 |
|-----|------|---------|
| 文本 | 服务名称、描述、介绍 | 营销语言与功能描述混杂 |
| 图像 | 截图、图标、封面图 | 视觉信息难以量化 |
| 数值 | 评分、销量、价格 | 数据可能缺失或噪声 |
| 行为 | 用户点击、停留、转化 | 隐性信息需要挖掘 |

理解一个服务，需要融合以上所有信息。

### 5.4.2 文本理解：从名称到描述

**服务名称理解**

服务名称通常简短（3-6字），但信息密度高：
- 核心功能词：外卖、打车、购物
- 品牌词：美团、滴滴、京东
- 修饰词：专业、便捷、私人

```python
# 服务名称解析示例
service_name = "美团外卖"

# 品牌识别
brand = "美团"  # 专有名词

# 功能识别
function = "外卖"  # 核心功能词

# 组合理解
meaning = brand + function = "美团公司的外卖服务"
```

**服务描述理解**

服务描述通常较长，包含更丰富的信息：

```
"美团外卖-您身边的美食专家。
支持外卖、跑腿、药店等多种服务。
新用户首单减10元，点击立即下单！"
```

这里需要提取：
- 功能：外卖、跑腿、药店
- 活动：新用户首单减10元
- 品牌定位：美食专家

### 5.4.3 图像理解：服务的视觉呈现

服务截图是理解服务的重要窗口：

- 通过截图判断服务类型（UI风格、功能布局）
- 通过截图评估服务质量（界面美观度、操作流畅度）
- 通过截图识别违规内容（敏感信息、广告导流）

```python
# 服务截图分类示例
from transformers import ViTForImageClassification

model = ViTForImageClassification.from_pretrained(
    'google/vit-base-patch16-224'
)

# 识别截图中的服务类型
# 餐饮类 → 食物照片为主，UI有菜单、购物车元素
# 出行类 → 地图元素，输入框布局
# 商家类 → 数字、表格、服务标识
```

### 5.4.4 多模态融合：综合判断

单一模态的信息往往不完整，需要融合多种信息：

**融合策略对比**

| 策略 | 做法 | 优点 | 缺点 |
|-----|------|------|------|
| 早期融合 | 特征拼接后统一编码 | 模态交互充分 | 计算量大 |
| 晚期融合 | 各模态独立预测后投票 | 独立优化 | 忽略交互 |
| 注意力融合 | 注意力机制动态加权 | 自适应权衡 | 复杂度高 |

**工业界实践**：

> 微信小程序UI自动化测试（CAT, 2024）的实践表明：
> - 纯LLM（0-shot）：50%完成率
> - 纯LLM（N-shot）：66%完成率  
> - 纯ML UI映射：52%完成率
> - 混合方法（RAG + ML）：**90%完成率**
> - 成本：仅0.34美元/次

这个案例说明：**在服务理解场景下，结合检索增强和机器学习的方法，往往比纯深度学习更有效**。

### 5.4.5 服务内容理解的前沿方向

**方向一：大语言模型的理解能力**

GPT-4、Claude等LLM展现出了强大的零样本理解能力：

```python
# LLM理解服务描述示例
prompt = """
分析以下小程序服务的描述：
{服务描述}

请提取：
1. 核心功能
2. 目标用户
3. 差异化特点
"""

response = llm.generate(prompt)
```

优点：理解深度强，可处理复杂描述
缺点：推理延迟高（秒级），成本高

**方向二：跨模态对齐**

CLIP等模型实现了文本-图像的语义对齐：

```python
# CLIP计算服务名称与截图的语义相似度
from transformers import CLIPProcessor, CLIPModel

model = CLIPModel.from_pretrained("openai/clip-vit-base-patch32")
processor = CLIPProcessor.from_pretrained("openai/clip-vit-base-patch32")

# 名称与截图是否匹配？
inputs = processor(text=[service_name], images=screenshot, return_tensors="pt")
similarity = model(**inputs).logits_per_image
```

### 5.4.6 批判性分析：理解的上限在哪里？

**问题一：体验的不可表达性**

服务的核心价值是体验，而体验是难以用文字和图片完全表达的。

"海底捞的服务好"——好在哪里？周到？热情？及时？这些体验无法从文字描述中完全提取。

**问题二：信息的不对称性**

服务提供者说的 ≠ 用户感知到的
- 宣传文案：极致体验
- 实际使用：卡顿、bug、客服难找

内容理解只能处理"显性信息"，隐性信息需要其他手段（如用户反馈分析）。

**问题三：多模态的融合之难**

不同模态的信息可能冲突：
- 文本说"品质保证"
- 截图显示"粗糙界面"
- 评价提到"经常出错"

如何权衡这些矛盾信息？目前没有完美解决方案。

## 5.5 服务质量评估：回答"服务好不好"

### 5.5.1 质量的多维性

"服务好不好"不是一个简单的问题。服务质量是多维的：

**功能维度**：服务是否提供了声称的功能？功能是否完整？

**性能维度**：响应速度如何？稳定性怎么样？

**体验维度**：界面美观吗？操作便捷吗？

**安全维度**：数据安全吗？隐私保护好吗？

**口碑维度**：用户评价好吗？评分高吗？

这些维度之间可能存在权衡：
- 功能强大但性能差
- 体验流畅但价格高
- 评分高但有安全隐患

### 5.5.2 质量评估的数据来源

**显性数据**：
- 用户评分（1-5星）
- 用户评价（文本评论）
- 投诉记录

**隐性数据**：
- 留存率（用户是否继续使用）
- 转化率（是否完成目标行为）
- 异常率（报错、闪退等）

**结构化数据**：
- 服务参数（响应时间、可用率）
- 资质认证（营业执照、行业许可）

### 5.5.3 质量评估的技术方案

**方案一：基于统计的评估**

```python
# 简单的统计指标
def calculate_quality_score(service):
    score = 0.0
    
    # 评分加权
    score += service.rating * 0.4
    
    # 销量加权
    score += min(service.sales / 10000, 1.0) * 0.2
    
    # 投诉率扣分
    score -= service.complaint_rate * 0.4
    
    return score
```

优点：简单、可解释
缺点：无法捕捉非线性关系

**方案二：机器学习评估**

```python
# 使用GBDT预测质量分
import lightgbm as lgb

features = [
    service.rating,
    service.sales,
    service.complaint_rate,
    service.response_time,
    service.update_frequency,
    # ... 更多特征
]

model = lgb.LGBMRegressor()
model.fit(X_train, y_train)

predicted_quality = model.predict([features])
```

**方案三：深度学习评估**

```python
# BERT + 回归
from transformers import BertModel

class QualityModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.bert = BertModel.from_pretrained('bert-base-chinese')
        self.regressor = nn.Linear(768, 1)
    
    def forward(self, input_ids, attention_mask):
        outputs = self.bert(input_ids, attention_mask)
        pooled = outputs.pooler_output
        quality = self.regressor(pooled)
        return quality
```

### 5.5.4 质量评估的特殊挑战

**挑战一：数据稀疏**

头部服务有大量反馈数据，长尾服务几乎没有。冷启动服务的质量如何评估？

**解决方案**：
- 迁移学习：从头部服务学到的知识迁移到长尾
- 特征补充：使用服务的基础特征（开发者资质、服务描述完整性）作为代理

**挑战二：维度冲突**

不同维度的质量可能冲突。"功能多"但"性能差"——算好还是不好？

**解决方案**：
- 多维评分：分别评估每个维度
- 用户分层：不同用户关注不同维度

**挑战三：时空变化**

服务质量不是固定的。促销期间可能变差，大促后可能恢复。

**解决方案**：
- 时序建模：引入时间序列特征
- 实时监测：基于流式数据快速更新

### 5.5.5 批判性分析：质量评估的哲学问题

**问题一：谁的质量？**

质量是主观的。老年用户和年轻用户对"好服务"的定义可能完全不同。评估体系代表的是谁的价值判断？

**问题二：短期还是长期？**

一个服务今天评分高，可能是"蜜月期"效应。用户长期留存才是真正的质量指标。

**问题三：可解释性的缺失**

深度学习模型能给出质量分数，但无法解释"为什么"。运营人员需要知道问题在哪里，才能改进。

**问题四：刷分与反作弊**

评分可以被操纵。如何识别虚假评价是持续的技术挑战。

## 5.6 服务知识图谱：结构化组织服务知识

### 5.6.1 什么是服务知识图谱？

服务知识图谱是**结构化的服务知识库**。

它用"实体-关系-实体"的三元组形式组织知识：

```
(美团外卖, 提供方, 美团公司)
(美团外卖, 属于, 外卖服务)
(美团外卖, 适用于, 餐饮场景)
(美团外卖, 替代服务, 饿了么)
(美团外卖, 互补服务, 美团单车)
```

这些知识从哪里来？从前面几节的服务分类、内容理解、质量评估中提取。

### 5.6.2 服务知识图谱的Schema设计

**核心实体类型**

| 实体类型 | 示例 | 属性 |
|---------|------|------|
| 服务 | 美团外卖、滴滴出行 | 名称、描述、类目、评分 |
| 服务商 | 美团公司、滴滴公司 | 名称、资质、规模 |
| 服务类目 | 外卖服务、出行服务 | 名称、父类目 |
| 场景 | 用餐场景、出行场景 | 名称、特征 |
| 用户群体 | 白领、学生、家庭 | 名称、特征 |

**核心关系类型**

| 关系 | 说明 | 示例 |
|-----|------|------|
| 属于 | 服务→类目 | (美团外卖, 属于, 外卖服务) |
| 提供 | 服务→服务商 | (美团外卖, 提供, 美团公司) |
| 适用于 | 服务→场景 | (外卖服务, 适用于, 用餐场景) |
| 面向 | 服务→用户群体 | (外卖服务, 面向, 上班族) |
| 替代 | 服务→服务 | (美团外卖, 替代, 饿了么) |
| 互补 | 服务→服务 | (外卖服务, 互补, 配送服务) |

### 5.6.6 批判性分析：知识图谱的局限性

**问题一：构建成本**

高质量的知识图谱需要大量的人工标注和审核。自动化构建的准确率有限，人工维护成本高。

**问题二：时效性挑战**

服务变化快，知识图谱如何及时更新？增量更新策略可以缓解，但无法完全解决。

**问题三：覆盖度问题**

长尾服务、新兴服务的知识难以获取。图谱的不完整性会影响应用效果。

**问题四：知识的价值验证**

构建了知识图谱，如何证明它带来了业务价值？需要设计合理的A/B测试来验证。知识图谱的价值往往体现在间接效果（如召回率提升）上，难以直接量化。

**问题五：服务边界的动态性**

小程序服务的内容经常变化，服务边界定义困难。如何动态更新知识图谱的Schema和实体是一个持续挑战。

## 5.7 本章小结

### 核心要点回顾

本章围绕"服务理解"这一核心命题，系统讲解了四大能力和一个基础设施：

1. **服务分类** — 回答"这是什么服务"
   - 建立服务与类目的映射
   - 技术：规则 → ML → 深度学习
   - 挑战：多标签、动态类目

2. **意图理解** — 回答"用户想要什么"
   - 意图识别 + 槽位填充
   - 技术：分类模型 → 联合模型 → 提示学习
   - 挑战：多意图、动态演化

3. **服务内容理解** — 回答"服务提供什么"
   - 文本 + 图像 + 多模态融合
   - 技术：BERT/ViT → CLIP → LLM
   - 挑战：体验不可表达、信息不对称

4. **服务质量评估** — 回答"服务好不好"
   - 多维质量评估
   - 技术：统计 → GBDT → 深度学习
   - 挑战：数据稀疏、维度冲突

5. **服务知识图谱** — 结构化组织以上知识
   - 实体-关系-三元组
   - 构建：识别 → 抽取 → 融合
   - 应用：搜索、推荐、问答

### 服务理解 vs 商品理解

| 维度 | 商品理解 | 服务理解 |
|-----|---------|---------|
| 本质 | 属性提取 | 能力理解 |
| 边界 | 清晰 | 模糊 |
| 表达 | 结构化参数 | 营销文案+体验 |
| 变化 | 慢 | 快 |
| 评估 | 客观指标 | 主观体验 |

### 前沿展望

1. **大语言模型服务理解**：LLM的推理能力将深度改变服务理解的方式
2. **多模态融合深化**：文本、图像、视频的统一理解
3. **实时理解系统**：流式数据驱动的持续学习
4. **个性化理解**：因人而异的服务价值判断
